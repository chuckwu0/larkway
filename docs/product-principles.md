# Larkway 产品原则与 Agent Team 设计基线



> 本文件记录 Larkway 的核心功能定义、产品原则与技术实现原则,作为所有产品 / 技术设计的基线(北极星)。活文档,随决策更新。

> 最后更新:2026-07-02(来源:owner 与 Elon 的对齐讨论)



## 一、核心功能:Agent Team



Agent Team(多 bot 团队协作:Elon 总协调 / Jobs 产品 / Linus 研发 / Coco 营销 / Turing 独立验收)是 Larkway 的核心 feature 与产品定义目标,按一等公民对待,不是附加能力。团队通过飞书群协作,交接需稳定、可追踪、断点可自愈。



## 二、产品原则



1. 聊天串是「视图」,不是「数据库」。工作的真相(状态 / 账本)与工作的通知(@ / 卡片)分离:通知尽力而为、可丢;状态持久、可查、可对账。

2. thin bridge。Larkway bridge 只传递场景 / 指针,不内置业务工作流;编排、超时检测、重试、重派、升级属于协调层(Agent workspace)逻辑,不把 bridge 做成工作流引擎。

3. @ 轻轨保留。@ 是自然、轻便、飞书原生的默认协作方式;只有复杂 / 长链 / 高价值任务才「升舱」到状态驱动的重轨。轻重双轨,按明确规则升舱。

4. 不破坏既有产品原则。任何新方案都要显式说明如何贴合以上原则。



## 三、技术实现原则(基于飞书底座)



1. 最大化复用飞书已有能力,尤其服务端能力:多维表格(Base)、原生任务、事件订阅、卡片回调等 server 侧现成能力优先;不自建重后端。

2. 最大化复用 Agent 底座能力:Claude Code / Codex 的定时(cron)、scheduled agent、workflow、session / memory;驱动层 / 对账 reconciler 可直接跑在 Agent 运行时上。

3. 状态存飞书、驱动靠 Agent 底座:优先让「状态账本」落在飞书(复用服务端),「驱动 / 自愈」落在 Agent 运行时(复用 Agent 底座),两边都不自建重后端。



## 四、进行中的专项(现状,随进展更新)



Agent Team 协作交接可靠性:产品调研(Jobs)+ 技术可行性调研(Linus)进行中。当前方向:轻重双轨 + 飞书 Base 任务账本 + Agent 运行时对账 reconciler;@ 降级为通知,卡片按钮回写状态。详细 PRD / 技术方案见后续产出。
